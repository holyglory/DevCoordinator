# DC-2026-07-18-INVITES-01 — Exact invite approval and user-owned Telegram notification bots

## Decision

Split verified identity from authorization. A successful Google OIDC callback may establish only the signed identity needed to render and submit an invite request. Every protected Console, route, API, proxy, and WebSocket request continues to recheck the current exact grant. The request target is derived from the authenticated host (`console` or the exact routed slug), never accepted from browser input. One pending email/resource request is idempotent; configured owners alone list and approve or deny incoming access requests. Approval creates or merges the exact grant and resolves the request in one private atomic access-policy write.

A Console-authorized account may register a Telegram bot and becomes that bot's owner. It may view and mutate only its own bot, project assignments, and Telegram authorization queue; configured Console owners may administer all bots. Tokens are checked with Telegram `getMe`, never returned by an API or written to logs, and retained only in a Console-owned mode-0600 state file because the running poller must recover them after restart. Project assignments store normalized coordinator `repo_id` values and derive labels from current inventory. Telegram users enter a bot-specific pending queue only after sending `/start` in a private chat. Approval and denial are separate from Google/Console access.

Use the Telegram Bot API's long-polling `getUpdates` path, with an offset committed after idempotent update processing. Registration inspects `getWebhookInfo`; it refuses an existing webhook unless the registering user explicitly takes it over, then calls `deleteWebhook` without dropping pending updates. Delivery uses `sendMessage`, a durable event cursor and per-recipient outbox, bounded exponential retry, Telegram `retry_after` handling, and permanent disabling of a recipient only for an explicit non-retryable Telegram rejection. The coordinator event journal—including events emitted for explicit host-observation state transitions—is the source, so activity initiated by any enrolled user or agent and detected crashes share one host-wide provenance boundary.

## Alternatives considered

- Keep rejecting unknown Google identities at callback: secure but cannot support a trustworthy request journey and causes sign-in loops on denied resources.
- Automatically grant access after Google verification: violates explicit per-resource approval and the configured-owner recovery boundary.
- Accept requested email, slug, or project from the client: allows target spoofing; server-derived identity/resource and immutable coordinator IDs are required.
- Let only configured owners register bots: simpler, but unnecessarily contradicts the user's request that Console users can register bots; bot ownership isolation preserves least privilege within the Console's existing broad operational grant.
- One global bot/token or name-derived project filters: mixes credentials and subscribers and repeats earlier project-attribution failures.
- Telegram webhooks: efficient, but require another public ingress and mutually exclude long polling. Long polling is operationally simpler for this private Console. An existing webhook is never removed silently.
- Send notifications directly from Console lifecycle button handlers or inventory diffs: misses Codex, Claude, other accounts, CLI actions, and durable crash provenance.
- Best-effort fire-and-forget delivery: loses notifications across restarts, Telegram rate limits, and transient network failures.

## Security and operational consequences

- A verified identity cookie is not an access grant; current policy remains the authorization source on every protected request.
- Invite submission is same-origin POST-only, rate/bound limited, and exact-resource idempotent. Resolved requests are retained only within a bounded audit window.
- Telegram tokens remain recoverable secrets on the Console host. Mode 0600, owner/type checks, atomic replacement, API redaction, and log redaction are mandatory; host root and the Console service account remain trusted.
- Only private Telegram chats are accepted. Telegram IDs are stored as integer-safe decimal strings, because Telegram documents identifiers with more than 32 significant bits.
- The event cursor and outbox are private durable state. A delivery is marked sent only after a successful Telegram response; replay after a crash is idempotent at the local event-recipient boundary.
- Source can be validated without a real Telegram credential through a deterministic fake Bot API. Production activation still requires the controlled Console/coordinator restart gate already recorded in `CompletionLedger.md`.

## Verification contract

- Unknown verified Google account receives an identity session but no protected access, can request only the host-derived resource, and cannot inspect or decide the queue.
- Duplicate, forged-resource, forged-email, cross-origin, malformed, over-limit, approval, denial, route rename/removal, and current-grant revocation paths are covered.
- Invited Console users can manage only bots they own; configured owners can manage all. Tokens never appear in list, error, log, or browser responses.
- Fake Telegram tests cover token validation, existing-webhook refusal/explicit takeover, restart-safe update offsets, private `/start`, duplicate updates, per-bot approval/denial, exact project fan-out, retry-after, transient/permanent send errors, and outbox replay.
- Coordinator tests prove a real server/Docker transition emits once, unchanged observations emit nothing, unobservable/unavailable samples do not invent a transition, and paginated event reads cannot skip committed events.
- Wide and narrow Console journeys verify the denied invite action, incoming invite queue, Telegram bot registration/assignment, Telegram authorization queue, populated/empty/error states, and destructive confirmations.
