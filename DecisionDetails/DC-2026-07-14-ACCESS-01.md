# DC-2026-07-14-ACCESS-01 — Per-account domain grants

## Requirements and evidence

The user requested an editable list of Gmail users and control over which
particular domain-assigned servers each user may access, including the DevOps
Console. The existing implementation had one environment-only
`ALLOWED_EMAILS` set. Every listed account could open the Console and every
`auth: "google"` route, while `auth: "public"` routes intentionally required
no identity.

The existing architecture also requires verified Google OIDC email claims,
immediate allowlist rechecks on every request, a domain-wide signed session,
default-private routes, identical HTTP/WebSocket enforcement, zero third-party
Node dependencies, and operator-controlled local state.

## Options considered

### Selected: configured owners plus local per-domain grants

`ALLOWED_EMAILS` remains the owner/recovery boundary. Owners always have the
Console and every route and are the only accounts allowed to edit access. A
new private state file holds invited Google account emails and grants for the
Console or exact assigned domains. OIDC and the existing session mechanism
remain unchanged; authorization is re-evaluated against current state on every
HTTP and WebSocket request.

This satisfies per-user and per-server/domain access without adding a service,
package, network dependency, or second sign-in flow. It preserves current
production owner access during migration. The important limitation is that
owner changes still require editing the private environment file and restarting
the Console. That is deliberate break-glass behavior, not an editable role.

### Rejected: all Console users administer access

This is simpler because a Console grant would imply access-administration
rights. It fails least privilege: any invited operator could grant themselves
other domains, invite new users, or remove peers. Recovering from an accidental
or hostile last-user removal would also require manual state repair.

### Rejected: one global editable allowlist

This matches the old model but cannot express the requested per-server grants.
Adding server lists as UI-only metadata would be misleading because the proxy
would still authorize globally.

### Rejected for this scope: Google Workspace groups or an external IAM proxy

External groups are useful when an organization already delegates identity and
group lifecycle to Workspace. An external identity-aware proxy can add richer
policy and audit features. Both introduce another operational control plane,
configuration and failure surface, and neither maps dynamic local route targets
without synchronization plumbing. Personal Gmail accounts also may not share a
managed Workspace lifecycle. These remain reversible future migrations because
the local policy has explicit users and resources rather than being embedded in
session cookies.

## Data and enforcement contract

- `access-control.json` lives under the private Console state directory and is
  written atomically with mode `0600`.
- Owners come only from `ALLOWED_EMAILS`; invited users come only from the
  access store. Both are normalized to lowercase. A stored invitation becomes
  usable only when verified Google OIDC returns that exact email address.
- Resource IDs are `console` and `route:<slug>`. The UI labels route resources
  with their real host and target. A server/container subdomain rename moves
  grants to the new hostname; deletion clears grants; creating a reused slug
  clears any stale grant before the route becomes reachable.
- Owners bypass resource grants. Invited users must have the exact resource.
  Public routes bypass identity as before.
- User removal makes an existing session unknown. Grant removal leaves the
  session valid for other resources but denies the removed resource on its
  next request.
- Only owners can read or mutate the full access list. A Console-granted invited
  user may operate the existing Console but cannot inspect or change access
  policy.
- Corrupt policy state is preserved for inspection and fails closed to owners
  only; it never restores invited access from untrusted data.

## Verification contract

Unit coverage must prove validation, owner immutability, atomic durability,
write-failure rollback, corrupt-state fail-closed recovery, concurrent deltas,
and immediate authorization decisions. End-to-end coverage must prove an
invited user can reach only a granted protected domain, cannot reach the
Console or another private domain, gains and loses access without a new cookie,
cannot administer access after a Console grant, and is fully revoked when
removed. HTTP and WebSocket paths must apply the same grants. Desktop and phone
browser verification must exercise the member list and add-user surface.
