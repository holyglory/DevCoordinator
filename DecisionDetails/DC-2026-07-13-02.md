# DC-2026-07-13-02 — DevOps Board source ownership is anchored to the login account

Original record: 2026-07-13 - DevOps Board source ownership is anchored to the login account

## Supporting record

Authority supersession: [DC-2026-07-15-HOST-01](DC-2026-07-15-HOST-01.md)
supersedes this record's claim that different effective UIDs retain separate
authorities and that the Board may aggregate them. On a managed host, every
enrolled account and agent uses the single service-owned authority. The login
account's home remains the correct location for that account's private launch,
log, migration, and reconciliation evidence, and remapped application `HOME`
values still must not redirect those files.

Decision: The Board and coordinator CLI use the effective POSIX account home,
resolved independently of a host application's runtime `HOME`,
`CFFIXED_USER_HOME`, or remapped user-domain root. This makes same-UID Codex
instances share one private evidence home and lock. An explicit isolated-mode
coordinator-home override remains authoritative only within that deliberately
non-host-global mode. Different effective UIDs retain private evidence homes,
but host authority, inventory, leases, and mutations converge through the
peer-authenticated system broker; one cross-user writable state directory is
forbidden. Launch readiness binds every automatic source identity to a
packaged-helper server count; a successfully decoded source is not by itself
proof that the intended records reached the UI.

Why: When Parall launched the signed Board, it remapped `HOME` and
`CFFIXED_USER_HOME` to a Parall-managed runtime home.
`FileManager.default.homeDirectoryForCurrentUser` followed that remapping, so
automatic discovery selected a valid runtime coordinator source with zero
managed servers. The login account's `~/.codex/agent-coordinator` contained 16
managed servers. The same user-home assumption also owned the default Manage
Sources configuration, so the defect could split both discovery and persistent
configuration across host-specific runtime trees. The earlier 1 MiB inventory
limit was a real independent defect, but fixing it did not establish source
ownership. The subsequent `loaded=1 total=1` launch result was therefore
incorrectly reported as proof that the real coordinator source was active; the
user's zero-server screen contradicted that claim.

Result: The prior launch evidence is reclassified as insufficient. The product
repair, legacy configuration migration, interprocess configuration transaction,
canonical source identity, same-UID coordinator convergence, per-source launch
evidence, and narrow-pane layout guard are implemented. The clean signed app
loaded all three discovered sources with server counts 0, 16, and 3 and rendered
all 19 managed servers. Four refresh cycles retained those counts, with the app
returning to idle and zero child processes between bounded inventory bursts.
