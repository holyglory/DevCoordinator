# DC-2026-07-18-SERVERS-01 — Server groups disclose immutable canonical membership

## Context

The authenticated Console Servers page showed 102 web-serving rows under a literal `other` header. The original projection incorrectly treated retained Docker history as current inventory: the store held 156 old `gf-v2-dev-*` rows, including obsolete short-ID aliases, but none of those resources was present in the latest completed Docker-available observation. The observer also required an exact enrolled root or a `.git` marker at a nested Compose working directory, so a current container launched from a path such as `/home/example/GlobalFinance/deploy` could remain unattributed even though `/home/example/GlobalFinance` was the enrolled canonical worktree. The Console correctly refused to infer ownership from the `gf-*` name, but the compatibility projection made absent history look like a large live fallback group.

The user also selected project headers and requested collapsible blocks. The existing flat cross-project pager made the complete active slice look like one long project and did not give each group a disclosure boundary.

## Decision

- Treat a Compose working directory as authoritative path evidence and resolve it to the deepest enrolled canonical worktree that contains it. A nearer nested `.git` marker remains a distinct repository; a name resemblance never establishes ownership.
- Request exact 64-hex Docker IDs from both `docker ps` and `docker stats`. Parse valid bulk-inspect output even when another container races away; an incomplete inspect preserves earlier labels, ports, health, restart policy, membership, and binding provenance instead of converting unknown evidence to absence.
- Define current Docker presence from the newest completed observation snapshot whose Docker capability is available. Keep all historical resource and observation rows, but exclude resources absent from that snapshot from compatibility inventory, current lifecycle and project-usage projections, and current unassigned-resource counts. A failed or Docker-unavailable observation does not manufacture an empty current inventory; the last completed Docker-available snapshot remains the presence boundary until a newer available snapshot commits.
- Keep retained short-ID records as history, but omit a 12-character compatibility row only when exactly one newer same-engine 64-hex identity is its strict prefix expansion and attribution/lifecycle evidence does not contradict the merge. Ambiguous or malformed candidates remain visible.
- Publish `container_resource_ids` in `project_usage`. Console grouping prefers those immutable IDs and uses container names only for older projections that omit the field entirely.
- Render every nonempty Servers project/resource group as a full-width native disclosure button. Groups start closed; opening one closes the prior group; disclosure is transient across a browser reload; only the open group's member page is mounted, capped at 75 resources. Honest unresolved evidence is labeled `Unassigned Resources`.

## Alternatives rejected

- A `gf-*` or `gf-v2-dev-*` UI rule was rejected because naming is neither unique nor mutation authority and would disagree with whole-project actions.
- Attaching retained `gf-v2-dev-*` rows one by one was rejected because those rows were absent history, not current host resources; doing so would conceal the presence-projection defect. Manual attachment is also not a durable substitute for canonical Compose attribution when a current container is replaced.
- Deleting the 12-character rows was rejected because their provenance and historical observations remain useful; strict compatibility suppression preserves them without double-presenting a physical identity.
- Leaving all groups expanded, allowing several groups open, or paging the flat cross-project list was rejected because each recreates the long mounted surface or hides project headers behind pagination.
- Persisting disclosure in account preferences or browser storage was rejected because stale cross-session state can conceal newly relevant operational resources; in-memory state survives polling and SPA navigation without becoming durable policy.

## Guard evidence

Focused must-catch tests first reproduced: missing `--no-trunc` identity and telemetry commands; bulk-inspect downgrade to short IDs; nested Compose paths failing canonical membership; a real nested worktree being collapsed into its outer repo; non-Git evidence mislabeled as conflicting claims; degraded inspection erasing attribution/ports/labels/health/restart policy; absent history leaking into current compatibility, lifecycle, project-usage, and unassigned projections; ambiguous or malformed alias suppression; name-based co-claiming in the browser; flat Servers pagination; and missing disclosure ARIA/exclusive state.

The corresponding false-positive controls retain legacy name matching only when an ID field is absent, keep ambiguous aliases visible, preserve a stopped resource that is actually present in the current snapshot, preserve the last available presence boundary across a Docker-unavailable observation, allow a later exact identity to reappear, preserve distinct nested repositories, and keep all project headers discoverable while paging only the selected group.
