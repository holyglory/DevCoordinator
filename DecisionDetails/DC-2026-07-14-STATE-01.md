# DC-2026-07-14-STATE-01 — Repository removal is a reversible decommission backed by the normalized coordinator store

Original record: 2026-07-14 - Repository removal is a reversible decommission backed by the normalized coordinator store

## Supporting record


Decision: Implement the normalized private SQLite/WAL coordinator store,
transactional same-UID legacy import, one observer per host resource domain,
and an authenticated Unix-socket broker wherever different effective users
share host-global ports or Docker resources. Repository installation state is
separate from repository identity. Removing a repository creates a durable
disabled fence before external work, disables every registered automatic-start
policy, stops exact owned resources, releases active leases, deactivates port
assignments, verifies the stopped boundary, and only then removes the repository
from active Board inventory. Files, containers, volumes, databases, backups,
and audit history remain. Reinstallation is available only through the
Coordinator skill and does not itself start the runtime. Unassigned resources
remain separate host resources and require an explicit attach or standalone
retirement decision; names never establish ownership.

Owner context: The owner was shown three plain-language choices on 2026-07-14.
A Board-only Hide plus the existing Stop was described as days of work but able
to lie because another agent/source or Docker policy could restart and
rediscover the project. A `removed_projects` list in each existing JSON home was
described as weeks of work but vulnerable to disagreement and stale
resurrection across the three coordinator homes, while adding migration debt.
The recommended normalized transactional store and authenticated cross-user
broker were described as a multi-month migration with substantial rollback,
fault-injection, and service-deployment complexity, but the only option that
can provide one repository identity, a durable future-start prohibition, safe
reinstallation, and an honest host-wide guarantee. SQLite/WAL was explained as
the private embedded database for one account or broker service; the broker was
explained as the OS-authenticated authority required only when different users
share host resources. The owner replied `yes`, approving the recommended option
and its reversible, data-retaining removal semantics.

Why: At approval, the coordinator reconstructed projects from denormalized
`state.json` records and had no repository installation/tombstone entity.
`project stop` did not fence future start/register/Compose/lease paths or
disable Docker restart policy. The Board therefore could not both hide a
project and truthfully promise that it remained stopped. The reported unassigned
`kosttracking-prod-copy-pg` container also demonstrated that repository
membership and host-resource control are different facts: name-only evidence
must not authorize mutation, but a uniquely controlled standalone resource
still needs an explicit retirement journey.

Result: The normalized store, transactional same-UID migration, lifecycle
fence, broker boundary, v2 Board graph, reversible repository removal, and
rollback paths are implemented. Release acceptance remains pending until the
original packaged-app journeys and the normal/optimized, native, remote Linux
cross-UID, repository-boundary, and freshness gates pass.
