# DC-2026-07-13-06 — Board inventory transport is bounded, compact, and launch-verified

Original record: 2026-07-13 - Board inventory transport is bounded, compact, and launch-verified

## Supporting record


Decision: DevOps Board inventory commands have a dedicated bounded 16 MiB
output allowance while every ordinary coordinator command retains its 1 MiB
limit. The Board requests compact JSON with at most the newest 30 Docker
history samples per primary container; the coordinator still persists and
returns 120 samples by default for other CLI and API consumers. Inventory is
decoded once in a task-group worker and handed to the main actor as a Sendable
value graph. A source transport failure leaves Docker capability unknown; the
Docker-unavailable warning is shown only after a source loaded evidence that
Docker itself is unavailable. The signed-app `--verify` path now tails a fresh
unified-log capture, binds the exact executable and process start identity,
requires at least one loaded source plus a stabilization window, and performs
bounded identity-safe cleanup while retaining private diagnostics on failure.

Why: The user's active coordinator emitted a valid 3,271,993-byte inventory,
but `SystemCommandExecutor` applied the ordinary 1,048,576-byte cap to it,
terminated the successful helper, and passed a truncated failure to the store.
The Board consequently stayed in initial loading, then labeled the coordinator
and every dependent capability unavailable, including a misleading Docker
warning. Each failed retry still paid for Docker observation and state work.
The prior July 13 build/run decision claimed a repeatable process-verification
surface, but its gate only slept one second and checked `pgrep`; the 69 native
tests used small fake JSON and the non-native repository gate used source-text
anchors. This contradicted the intended operational meaning of “Board is
running”: an alive signed process was treated as delivery success without
proving that its real source completed. The earlier caching, visibility, and
30-second idle-interval decisions remain valid, but could not compensate for a
first snapshot that was discarded at the transport boundary.

Result: The exact live Board request is now 650,020 bytes and completes in
2.86 seconds with Docker available, 16 servers, 15 containers, seven PostgreSQL
projections, and 450 returned history samples. The signed app built from code
commit `7d77304d1925080a25d2f11465d5599f30d6265b` passed the new real-source
launch gate; two independent provenance-bound launches emitted four and then
three consecutive `loaded=1 total=1` completion markers. Process sampling
observed 0% idle CPU, a brief 17.3% refresh sample, roughly 116-124 MiB RSS,
and no persistent coordinator child. Native tests now
cover the realistic over-1-MiB failure shape, ordinary-cap preservation,
single-decode handoff, and capability truth; coordinator tests cover compact
formatting, limits 0/30/120, current-stat preservation, out-of-range inputs,
chronological concurrent history, and hostile executable overrides; the launch
detector has must-catch and false-positive controls for source failure, stale or
reused process identity, split log writes, capture/app death, partial success,
stabilization, and TERM-to-KILL cleanup.
