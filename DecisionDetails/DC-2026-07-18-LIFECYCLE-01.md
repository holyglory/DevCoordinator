# DC-2026-07-18-LIFECYCLE-01 — Archive is reversible decommission; removal is exact typed cleanup

## Context

The Coordinator already has a repository lifecycle named `remove`, but its implemented and recorded contract is a reversible decommission: fence new starts, capture and disable auto-start policy, stop and verify exact managed resources, release active allocations, hide the repository, and retain its files, containers, volumes, databases, backups, and audit history. The Console also has a separate per-browser Hide preference which does not stop or fence anything. The requested cleanup tools need both reversible Archive and permanent Remove without making either of those older behaviors lie.

One canonical Git worktree is one Coordinator project. A project catalog entry, a local worktree, a Git common object directory, a Docker container, a managed server definition, and a remote repository are separate targets with different ownership and recovery boundaries.

## Decision

- `repository plan-remove`, `repository remove`, `repository list-removed`, and `repository reinstall` retain their existing semantics as compatibility aliases for the canonical `plan-archive`, `archive`, `list-archived`, and `restore` journey. Restore only clears the durable fence and restores captured policy; it never starts a runtime.
- Project permanent removal is a catalog operation after successful Archive. It removes the project from active Coordinator catalog, allocation, and control projections while retaining an immutable tombstone and audit history. It never deletes a checkout, Git objects, containers, images, volumes, databases, backups, logs, credentials, networks, Console route or Gmail-grant configuration, or a remote repository. Console routes and grants remain separate cleanup targets because their current durable identity is not bound to an immutable Coordinator project ID; name-based cascade deletion could remove access to a later replacement.
- Managed-server Archive is an exact server-specific start fence plus captured policy disablement, verified stop, lease release, assignment deactivation, and removal from active target resolution. Restore clears only that exact fence, replays only captured policy, and never starts the server. Permanent removal is allowed only from archived, verified-stopped state; it deactivates the definition and assignment while retaining audit evidence and any log reference. Log deletion is a separate operation and is permitted only for a proven Coordinator-owned private log path.
- Docker Archive binds the full immutable container ID, engine identity, controller, and current observation; disables restart, stops, verifies, and hides the exact container while retaining its configuration and data. Permanent removal runs the equivalent of `docker container rm <full-id>` only for an archived, stopped, identity-matching container. It never uses force or `-v` and never removes images, volumes, databases, networks, or sibling containers. Compose-owned resources require an exact enrolled logical service/stack boundary and are refused when durable per-service disablement cannot be proved.
- Local worktree removal is distinct from project catalog removal. It is allowed only after project Archive and the separately confirmed project-catalog purge have left a non-restorable project tombstone, and only for a proved secondary linked worktree, with exact path, device/inode, Git common-directory/worktree identity, HEAD, branch, and clean-status evidence. This ordering prevents removing the physical source while the project still promises Restore. It refuses the primary worktree, dirty tracked files, untracked or ignored files, submodules, locks, symlink components, nested mounts, unavailable process/mount observation, active processes or open files, active resources, detached/unanchored state, and identity drift. It uses `git worktree remove -- <path>` without force and verifies both the directory and administrative entry are gone. Primary-clone deletion and remote-repository deletion are excluded.
- Every archive or removal is a two-step durable plan/apply transaction. The plan records the operation kind, actor, reason, exact immutable target identities and controller fingerprints, observable policies and allocations, effect/retention lists, blockers, confirmation phrase where destructive, and a SHA-256 fingerprint. Apply rechecks live authorization and fresh host/Git/Docker/process truth, refuses drift before mutation, writes the fence before external effects, records every phase, resumes idempotently after interruption, and preserves partial/needs-attention evidence.
- Console permanent removal is initially restricted to configured Console owners. Browser-visible names are presentation only; API requests and plans bind opaque Coordinator identities. The existing browser Hide preference remains separate and is never labeled Archive.
- Archived and removed records are first-class retained data, excluded from active projections, and available through an Archived collection. Removed physical targets offer recovery/recreation only when a real retained source supports it; they never promise Restore.

## Alternatives considered

- Repurpose `repository remove` as destructive deletion: rejected because it would silently break the established reversible, data-retaining contract and make existing callers unsafe.
- Relabel the Console Hide control: rejected because it is cosmetic, browser-local, and auto-reverses when a resource runs.
- Cascade-delete a project and everything underneath it: rejected because catalog membership does not establish authority to destroy Git objects, worktrees, persistent Docker data, logs, routes, credentials, or remote repositories.
- Use names, paths supplied by the browser, container short IDs, or current PIDs as deletion identity: rejected because they can be reused, replaced, ambiguous, or raced.
- Use `rm -rf`, `git worktree remove --force`, `docker rm --force`, `docker rm -v`, Compose `--volumes`, or image pruning: rejected because they discard unproven source or persistent data and cross independent lifecycle boundaries.
- Hard-delete normalized rows and history: rejected because it destroys attribution and makes crash recovery, replay, and no-resurrection projection impossible.

## Verification contract

Tests must cover archive/restore/no-auto-start, permanent-removal confirmation, plan drift, revocation between plan and apply, idempotent replay, interruption at every external-effect boundary, active-versus-archived projections, exact Docker replacement/name reuse, running/paused/stopped and restart-policy states, Compose and database/volume refusal, server PID reuse/zombie/wrong-listener/unobservable identity, and worktree main/locked/dirty/untracked/ignored/submodule/symlink/mount/process controls. Console evidence must cover owner gating, exact target confirmation, failure/partial states, focus restoration, bounded active and archived collections, and wide and narrow viewports.
