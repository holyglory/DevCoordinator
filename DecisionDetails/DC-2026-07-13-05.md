# DC-2026-07-13-05 — Background inventory refresh stays cached and edge-triggered

Original record: 2026-07-13 - Background inventory refresh stays cached and edge-triggered

## Supporting record


Decision: DevOps Board keeps the 30-second external-truth reconciliation poll,
but treats it as background work once a source snapshot exists. Surface
visibility is now gated on the aggregate hidden/visible edge, so duplicate
window callbacks and menu-to-window handoffs do not restart polling. Opening
the menu no longer forces an additional refresh; the explicit Refresh control
remains available. While a scheduled refresh is running, the last proven
source/capability/resource snapshot stays visible until the replacement is
complete. The menu source summary is a full-row native Button rather than
link-styled inert text.

Why: Reproduction showed that every popover open forced an inventory command,
`setSurfaceVisible` re-requested work for unchanged/per-surface callbacks, and
every poll replaced healthy source state with `loading`. That made the menu to
console journey reliably show `Refreshing inventory` even when cached data was
usable. Unscoped inventory also performed a second Docker-backed pass solely
to add project backup directories, repeating the expensive
`docker stats --no-stream` observation.

Result: The backup-enrichment pass now uses `--no-docker` and merges only its
backup evidence into the first authoritative runtime snapshot. Deterministic
tests prove initial loading remains truthful, a blocked background refresh
retains nominal cached presentation, duplicate/overlapping visibility events
do not issue commands, a real hidden-to-visible edge still refreshes, and the
no-Docker enrichment retains the first pass's Docker telemetry even when
backup discovery fails, and combined Docker/backup failures retain both
operator-relevant reasons. A failed first load followed by a blocked retry also
proves that failed-without-cache state returns to truthful loading rather than
showing a completed empty inventory. The existing
source-presentation regression exposed a contradiction with the earlier
publish-on-change decision: `CoordinatorOrigin` identity equality intentionally
ignores label/state-path presentation changes, so an otherwise equal inventory
could retain an old source label. Inventory publication now also checks the
current source presentation, preserving low-churn publishing without hiding
real configuration changes.

Release packaging exposed the adjacent first-launch case: a failed normalized
host observation short-circuited the pure inventory read, so usable committed
state appeared unavailable. The load pipeline now always attempts the read,
keeps a successfully decoded source available, and presents the observation
failure as an actionable warning with retained diagnostics. A must-catch test
proves the failing observation is followed by an inventory read and that the
committed repository/server graph remains visible.

The subsequent signed launch reached that committed read and exposed a wire
contract defect: `AccountStore.inventory_v2()` built normalized lease and port
assignment rows, then a broad compatibility merge replaced those two
top-level collections with legacy `id`/`project`/`name` rows. The Board
correctly rejected the payload because schema v2 requires durable
`lease_id`/`assignment_id` and `repo_id` identities. Normalized collections now
remain authoritative at the top level, while their legacy shapes live only in
`v1_compatibility`; project scoping filters each representation by its own
identity model. Must-catch store and scoped-CLI tests use nonempty leases and
assignments, and the Board projection excludes released leases and inactive
assignments from its active operational view. This closes the fixture gap that
previously tested only hand-authored v2 JSON or empty colliding collections.
Console and production registration-readiness consumers now explicitly project
and validate the nested compatibility rows instead of guessing a schema from
colliding top-level names. Project-scoped inventory also filters every
repository-owned normalized resource, observation, lifecycle violation, event,
backup, and restore record; host-global observation snapshots are omitted
because they cannot be truthfully attributed to one requested repository.
Docker scope is the union of repository memberships, database bindings,
control bindings, and lifecycle-violation identities, so a missing membership
cannot make a required resource disappear or falsely enable whole-project
control. Only coordinator sources and Docker engines referenced by that scoped
graph remain visible; an unrelated healthy source or engine therefore cannot
mask the selected repository's unavailable authority. Disabled repositories
remain absent from the active repository list, but their exact running-resource
violation context stays present for corrective action. The nested legacy
projection closes over those retained normalized server and Docker identities
as well as ordinary project paths, because lifecycle-violation compatibility
rows intentionally carry a null project rather than pretending ownership.
Current URL rows follow the same retained server identities. The published
HTTP contract is now test-bound to the normalized query-only schema, nested
compatibility projection, and implemented relocate route. Board assignment
matching also follows SQLite's exact case-sensitive `(repo_id, server_name)`
identity, so valid `web` and `Web` definitions neither collide nor exchange
ports.

The clean signed launch gate then exposed a verifier mismatch rather than an
inventory failure: the packaged helper measured 18 canonical normalized server
definitions, while the Board truthfully presented those definitions plus one
distinct retained `missing_repo` server incident. The preflight previously
counted only the legacy compatibility `servers` collection. It now mirrors the
normalized Board projection by counting canonical definitions and distinct
unassigned/lifecycle server incidents, with a legacy fallback only for
pre-schema-v2 payloads. A must-catch detector probe proves the former
legacy-only count rejects this production-shaped 18-plus-one fixture, while
the corrected verifier measures 19 without double-counting an incident that
appears in both normalized diagnostic collections.
