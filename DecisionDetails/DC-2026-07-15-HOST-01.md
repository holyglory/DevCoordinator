# DC-2026-07-15-HOST-01 — The host coordinator is one peer-authenticated system authority

## Supporting record

Decision: Supersede the account-authority portion of DC-2026-07-13-02 and
DC-2026-07-14-STATE-01. A managed host has one service-owned normalized
Coordinator authority at `/var/lib/devcoordinator/coordinator.sqlite3` and one
client endpoint at `/run/devcoordinator/broker.sock`. Codex, Claude, desktop
applications, and other enrolled agents use the Unix socket by default. The
kernel peer UID plus an administrator-provisioned exact repository/resource
ACL authorizes every mutation. Authorized clients read the same host inventory.
Clients never open or write the service database.

The service database, WAL, backups, operation journal, leases, assignments,
repository fences, resource observations, and host inventory are the only
Coordinator authority. Per-account files may remain only as private migration
inputs, launch/log evidence, and reconciliation journals; they cannot allocate
a host-global port, define a repository's authoritative lifecycle, or supply
the Console inventory. An explicit isolated-account mode remains available for
deterministic tests and hosts that deliberately do not install the service, but
it is not the product default and must identify itself as non-host-global.

The canonical executable and agent skills remain direct absolute symlinks to
this repository so source changes do not create installed-copy drift. Symlinks
are a discovery/update mechanism only. They do not bypass filesystem
permissions. Installation gives clients read/execute access to the canonical
code and group access to the socket; it never gives them write access to the
database or repository source.

Why: The prior private-account default allowed two effective UIDs on one host
to create independent inventories and assignments. On 2026-07-15 the
`holygloryTT` Coordinator started the healthy `prtzn-vpn` server on port 3001,
while the Console consumed the `holyglory` authority and therefore could not
see it. The optional broker prevented some port and Docker races but did not
make server lifecycle or inventory host-wide, so it did not satisfy the
owner's explicit requirement that the Coordinator work across user accounts
and agents by default.

Options considered:

- A cross-user writable directory, possibly reached through per-home
  symlinks, was rejected. A symlink preserves the target's Unix permission
  checks, and granting several UIDs direct SQLite/WAL access loses the private
  ownership, peer identity, least-privilege ACL, and safe recovery boundaries.
- Keeping private account authorities and teaching every UI to merge them was
  rejected. It can improve visibility but still leaves multiple writers,
  conflicting port reservations, duplicated host observation, and source-bound
  mutations.
- One service authority behind a peer-authenticated Unix socket was selected.
  It costs more installation, migration, and cross-UID testing, but it is the
  only option that gives one reservation lock, one inventory, one durable
  repository fence, attributable cross-account access, and no recurring
  privilege prompt.

Operational contract:

1. The default client fails closed when the system profile/socket is absent,
   stale, unsafe, or inaccessible. It never silently creates a new per-user
   authority.
2. The broker validates socket ownership/mode, kernel peer credentials,
   database generation, enabled principal, exact repository/resource grant,
   and startup fence before mutation.
3. Server observations published by a client are accepted only after the
   service proves the exact process UID, canonical repository cwd, process
   identity, listener, and port. A failed authoritative publication rolls back
   a new start or leaves explicit reconciliation evidence after a stop.
4. Authorized clients and the Console read the same committed service-store
   inventory. Read visibility does not imply mutation authority.
5. Installation is an explicit root transaction with private backup and
   rollback evidence. It migrates legacy authorities only after quiescing
   their writers, preserves conflicts instead of guessing, installs canonical
   symlinks, and verifies fresh-session discovery.
6. The daemon may inspect and stop exact cross-UID resources under its service
   authority. The narrow broker runs as root to match the administrator-owned
   store and inspect cross-UID resources, but service authority mode must reject
   every direct host-process workload-launch CLI surface. Host-process launches
   stay under the enrolled client UID or another explicitly recorded non-root
   runtime identity; typed Docker lifecycle remains broker-owned.

Required evidence: deterministic two-UID convergence, unauthorized-UID
rejection, exact repository ACL isolation, database-inaccessibility controls,
cross-UID start-to-inventory-to-Console visibility, concurrent global lease
serialization, symlink/source-update identity, migration rollback, broker
restart/recovery, and full repository validation.
