# DC-2026-07-15-HOST-01 — The host coordinator is one peer-authenticated system authority

## Supporting record

Decision: Supersede the account-authority portion of DC-2026-07-13-02 and
DC-2026-07-14-STATE-01. A managed host has one service-owned normalized
Coordinator authority at `/var/lib/devcoordinator/coordinator.sqlite3` and one
client endpoint at `/run/devcoordinator/broker.sock`. Codex, Claude, desktop
applications, and other enrolled agents use the Unix socket by default. The
kernel peer UID plus an administrator-provisioned exact repository/resource
ACL authorizes every mutation. Authorized clients read the same host inventory.
Clients never open or write the service database.

The service database, WAL, backups, operation journal, leases, assignments,
repository fences, resource observations, and host inventory are the only
Coordinator authority. Per-account files may remain only as private migration
inputs, launch/log evidence, and reconciliation journals; they cannot allocate
a host-global port, define a repository's authoritative lifecycle, or supply
the Console inventory. An explicit isolated-account mode remains available for
deterministic tests and hosts that deliberately do not install the service, but
it is not the product default and must identify itself as non-host-global.

Host and Docker sampling is a typed, authorized `HOST_OBSERVE` broker
operation, not a client-side database publication. The service claims one
observation owner per host, builds the snapshot against an explicit
service-owned `AccountStore`, and commits it atomically to the host authority.
Client-published managed-server registration remains a separate path whose
process, repository, listener, and port claims the service verifies before
acceptance. Read-only inventory never performs an implicit observation.

The canonical executable and agent skills remain direct absolute symlinks to
this repository so source changes do not create installed-copy drift. Symlinks
are a discovery/update mechanism only. They do not bypass filesystem
permissions. Installation gives clients read/execute access to the canonical
code and group access to the socket; it never gives them write access to the
database or repository source.

The base broker unit retains `ProtectHome=read-only`. Host-file lifecycle work
such as removing a proved clean secondary worktree receives no ad-hoc
elevation and no symlink permission bypass. Instead, the installation
transaction generates one replacement-scoped systemd drop-in that resets
`ReadWritePaths` and names only the canonical, directly-under-`/home` home
directories for the complete current enrolled-client set. Reapplying with a
smaller set removes omitted homes; a running broker keeps its prior mount
namespace until a safely drained restart. Exact peer, repository/resource ACL,
plan, fingerprint, identity, and filesystem checks remain mandatory inside
that writable namespace. `/home` itself, `/root`, `/etc`, and every unenrolled
home remain read-only.

Why: The prior private-account default allowed two effective UIDs on one host
to create independent inventories and assignments. On 2026-07-15 the
`holygloryTT` Coordinator started the healthy `prtzn-vpn` server on port 3001,
while the Console consumed the `holyglory` authority and therefore could not
see it. The optional broker prevented some port and Docker races but did not
make server lifecycle or inventory host-wide, so it did not satisfy the
owner's explicit requirement that the Coordinator work across user accounts
and agents by default.

Options considered:

- A cross-user writable directory, possibly reached through per-home
  symlinks, was rejected. A symlink preserves the target's Unix permission
  checks, and granting several UIDs direct SQLite/WAL access loses the private
  ownership, peer identity, least-privilege ACL, and safe recovery boundaries.
- Keeping private account authorities and teaching every UI to merge them was
  rejected. It can improve visibility but still leaves multiple writers,
  conflicting port reservations, duplicated host observation, and source-bound
  mutations.
- One service authority behind a peer-authenticated Unix socket was selected.
  It costs more installation, migration, and cross-UID testing, but it is the
  only option that gives one reservation lock, one inventory, one durable
  repository fence, attributable cross-account access, and no recurring
  privilege prompt.
- Keeping every home permanently read-only was rejected for explicitly
  authorized worktree removal because the broker could inspect but not remove
  the exact directory. Making all of `/home` writable or relying on symlinks
  was rejected because it crosses unenrolled accounts or preserves the same
  permission boundary. A transactionally replaced exact enrolled-home list
  preserves the systemd sandbox everywhere else and makes revocation
  deterministic on reapply plus restart.

Operational contract:

1. The default client fails closed when the system profile/socket is absent,
   stale, unsafe, or inaccessible. It never silently creates a new per-user
   authority.
2. The broker validates socket ownership/mode, kernel peer credentials,
   database generation, enabled principal, exact repository/resource grant,
   and startup fence before mutation.
3. Server observations published by a client are accepted only after the
   service proves the exact process UID, canonical repository cwd, process
   identity, listener, and port. A failed authoritative publication rolls back
   a new start or leaves explicit reconciliation evidence after a stop.
4. Authorized clients and the Console read the same committed service-store
   inventory. Read visibility does not imply mutation authority.
5. Installation is an explicit root transaction with private backup and
   rollback evidence. It migrates legacy authorities only after quiescing
   their writers, preserves conflicts instead of guessing, installs canonical
   symlinks, and verifies fresh-session discovery.
6. The daemon may inspect and stop exact cross-UID resources under its service
   authority. The narrow broker runs as root to match the administrator-owned
   store and inspect cross-UID resources, but service authority mode must reject
   every direct host-process workload-launch CLI surface. Host-process launches
   stay under the enrolled client UID or another explicitly recorded non-root
   runtime identity; typed Docker lifecycle remains broker-owned.
7. `HOST_OBSERVE` tickets retain the authenticated invoking UID and exact ACL
   decision through execution. The broker bounds queued work, rejects new work
   once shutdown begins, permits only the claimed host owner to fail, drain, or
   finalize its observation, and aggregates body plus cleanup failures in the
   returned structured error. Cancellation or broker shutdown cannot leave a
   second observer owner or a falsely completed snapshot.
8. The production host-snapshot callback contract is `AccountStore`, not the
   narrower broker-persistence store: it must expose host identity plus the
   normalized inventory projection used to reconcile the new observation.
   Tests exercise this exact callback through the real broker runtime and
   store-backed mutation backend rather than substituting an in-memory shape.
9. The installer derives writable home paths only from the complete explicit
   enrolled-client list, rejects noncanonical or indirect home layouts, resets
   the drop-in before writing the new set, and verifies the effective unit
   keeps unrelated homes and protected system paths read-only. Installation or
   rollback preserves exact prior bytes and mode; a live boundary change
   requires a safely coordinated broker restart.

## Live migration and incident evidence

The first live `HOST_OBSERVE` attempt after the 2026-07-18 rollout failed
closed. Operation `7a52222c-240a-4df7-ad8a-34bc5a51b227` reached the production
callback with a narrow `CoordinatorStore`, but the normalized snapshot builder
required `AccountStore.inventory_v2()`. The operation published no new
snapshot, and the invoking account's private database remained byte-for-byte
unchanged. A regression test then reproduced the exact production callback
boundary before the callback was changed to open the service database as an
`AccountStore`; the full broker and normalized suites pass in normal and
optimized Python.

The second attempt used verified backup
`/var/lib/devcoordinator-backups/20260718T151825Z-servers-grouping-host-observe-v3/service-20260718T151826Z-81d9574acd214ff98cadf1b062ea39a8.sqlite3`
with SHA-256
`b2d5320124d46bff97a81ed012995ea537c9441aae149470488ef66d7b9c603f`.
The backup manifest reports successful integrity, foreign-key, and coordinator
invariant checks. The corrected operation committed snapshot
`704279b3-9672-4f86-843e-aa21f1f54e69` at
`2026-07-18T15:25:59.055968Z` (observation revision 237) to the service store;
the client database hash again remained unchanged, and service-store inventory
matched the committed snapshot.

That restart also exposed a separate operational gap. Restarting the
coordinator dependency stopped Console, a coordinator API request transiently
returned HTTP 500 before recovery, and the restart interrupted a concurrent
PostgreSQL backup. The host observation itself recovered and committed, but a
production restart is not ready while it can race active broker work or leave
Console recovery dependent on one-shot startup timing. Quiescence and bounded
Console startup/recovery guards remain active completion work rather than being
misreported as part of the successful snapshot verification.

A later read-only cross-account lifecycle probe exposed a second contract gap.
UID 1001 could see the shared Console server but was not enrolled for its exact
repository/server ACL, so denial was correct; system-mode stop nevertheless
opened UID 1001's private reconciliation journal first and surfaced the
misleading internal error `matching server not found`. The client now resolves
the authenticated broker profile and exact repository/server enrollment before
any journal read, reservation, or signal, and rejects a visible-but-unenrolled
target with a typed broker-configuration/access result. This prevention guard
does not implement operational-contract item 6 for a second UID that *is*
explicitly enrolled: authoritative cross-UID stop/restart still needs a typed
broker lifecycle operation, live ACL reauthorization, service-store lifecycle
commit, and private-journal reconciliation, and remains in the completion
ledger.

Source-level restart hardening now fences every broker mutation atomically in
the SIGTERM turn, rejects queued or later mutations, retains accepted clients,
and waits for admitted handlers for a configured broker-side graceful budget.
The source unit pins `KillMode=mixed`, exact TERM/KILL behavior,
`SurviveFinalKillSignal=no`, no stop hooks or drop-ins, the Docker configuration
environment, and `TimeoutStopSec=65min`. A private install transaction migrates
only the exact known legacy Docker environment drop-in, preserves it for
rollback, and refuses ownership, mode, inode, content, or unrelated-directive
drift. Real isolated systemd 257 output supplies the effective-property fixture,
and deterministic tests cover signal-turn admission, queued rejection,
composite systemd/launchd policy interruption and replay, unit drift, and the
drop-in migration/rollback boundary.

That 65-minute setting is deliberately not recorded as a terminal database
guarantee. The outer broker helper timeout does not prove that its nested
`docker exec` client or in-container PostgreSQL process stopped, so a timed-out
dump or restore can remain outcome-uncertain after the broker handler returns.
Restore replay also has an independent idempotency gap: restore metadata is
committed before the broker operation is finished, and replay in that interval
can create another restore event and increment the restore count for one
physical restore. Both items remain active in the completion ledger. The
production unit has not been replaced with the source unit; its effective-unit
checker therefore correctly rejects the retained legacy drop-in, old
`KillMode`, and old timeout.

At `2026-07-18T16:36:33Z`, a separate task running from an external
`Skydive.Live` checkout restarted the production broker. The sudo journal
identifies that external working directory and command; the
Coordinator operation journal contains no operation spanning the restart.
Console and the API stopped as dependencies, then Console recovered on the
first systemd start and published its exact MainPID/lease/assignment graph after
five bounded readiness polls. This is useful recovery evidence, but it neither
installed the new unit nor exercised an admitted backup during shutdown and
therefore does not close the controlled in-flight restart acceptance item.

Required evidence: deterministic two-UID convergence, unauthorized-UID
rejection, exact repository ACL isolation, database-inaccessibility controls,
cross-UID start-to-inventory-to-Console visibility, concurrent global lease
serialization, symlink/source-update identity, migration rollback, broker
restart/recovery including dependent Console convergence and in-flight-work
terminal-state reconciliation plus idempotent replay, and full repository
validation.
