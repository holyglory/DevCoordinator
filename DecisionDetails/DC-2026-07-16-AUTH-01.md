# DC-2026-07-16-AUTH-01 — Google-protected routes translate private upstream credentials

## Supporting record

Production reproduction separated the two authentication boundaries. An
anonymous request to `https://prtzn.vr.ae/` returned the Console's 302 redirect
to Google login, while the route's exact loopback target returned HTTP 401 with
`WWW-Authenticate: Basic realm="prtzn-vpn"`. The saved `prtzn` route already
had `auth: "google"`; the browser prompt therefore came from the independently
protected application after Google succeeded, not from a route-policy change.

The host is explicitly multi-user. Removing prtzn's operator-token check would
let other local accounts call a privileged cloud/VPN control plane directly on
loopback. Trusting an unsigned forwarded-user header has the same flaw because
another local process can forge it. Keeping pass-through Basic authentication
would preserve two login journeys and allow caller-selected backend identity.
Adding a separate external identity-aware proxy would introduce another policy
and operational control plane for a dynamic local route that the Console
already authenticates and authorizes.

The selected bridge stores one optional backend credential per route in the
external private Console state directory, atomically at mode `0600`. Route and
API views expose only `{configured, scheme}`. On Google-protected HTTP and
WebSocket requests, the proxy removes caller `Authorization`, injects the
stored credential server-side, and removes `WWW-Authenticate` and
`Authentication-Info` from upstream responses. Public routes receive no stored
credential and preserve ordinary HTTP-auth request/response headers. Only
configured owners can rotate credentials through the same-origin Console API;
the deployment CLI accepts secrets only on stdin and never prints them.

The prevention test first reproduced the defect: a caller Basic header reached
the protected upstream instead of the configured backend credential. After the
change, focused proxy tests prove replacement and challenge suppression with a
public-route false-positive control. Private-store tests cover validation,
permissions, redaction, atomic persistence, concurrent mutations, write
failure, corruption, rename/delete lifecycle, and CLI non-disclosure. The real
TLS/OIDC full-stack test proves anonymous Google redirection, authenticated
credential translation, wrong-credential 401 without a browser challenge,
owner-only rotation, public-route rejection, and absence of the secret from
overview/API output.

Production deployment retained the failed manifest attempt at
`20260716T182155Z-prtzn-upstream-auth` and a separately timestamped, verified
rollback backup at `20260716T182211Z-prtzn-upstream-auth`. The installed
credential state is owned by the Console service account at mode `0600`; the
operator CLI reports only the `prtzn` slug and `bearer` scheme. After the
Console restart, systemd reported PID `3209284`, and server-wide coordinator
inventory proved that exact PID healthy on port 443 with the active durable
lease and assignment. A production TLS probe then proved anonymous `GET /`
still redirects to Google (302), an owner session receives the prtzn HTML app
(200) without an HTTP-auth challenge, and a caller Basic header cannot replace
the server-side credential (200 without a challenge). Direct loopback access
to the application remains independently protected with 401 Basic.
