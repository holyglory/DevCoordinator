# DevOps Board inventory readiness incident

## Outcome

Status: diagnosed

The Board and coordinator repairs are implemented and pass their deterministic/native suites. The final signed-app launch gate still has to prove the repaired source set and server counts before this incident is closed.

## Cause

Class: implementation
Confidence: confirmed
Request: DevOps Board must run against the user's real coordinator source, keep refreshes efficient, expose working source details, and truthfully show the managed servers owned by that source.
Immediate cause: Two defects occurred in sequence. First, the Board's coordinator wrapper applied a 1,048,576-byte command-output ceiling to a valid 3,271,993-byte inventory and discarded the truncated snapshot. After that limit was fixed, automatic source discovery still derived the user home from `FileManager.default.homeDirectoryForCurrentUser`; under the Parall launch environment, `HOME` and `CFFIXED_USER_HOME` remapped that value to a Parall-managed runtime home. That runtime home contained a valid coordinator source with zero managed servers, while the login account's `~/.codex/agent-coordinator` contained the expected managed-server records.
Why missed: Native tests injected explicit homes, and the launch detector accepted any stable app process that logged at least one syntactically loaded source. Its `loaded=1 total=1` marker carried neither the resolved source-home identity nor resource counts, so a valid but empty source under the wrong owner passed. The prior delivery therefore incorrectly described that marker as proof that the real coordinator source was loaded; the user's follow-up screen contradicted that claim.
Evidence: The original unavailable state reproduced the 1 MiB truncation against a 3,271,993-byte valid account inventory. For the remaining no-server state, the signed app process inherited Parall-remapped `HOME` and `CFFIXED_USER_HOME`; automatic discovery resolved the corresponding runtime coordinator source, whose inventory command succeeded with zero managed servers. Running the same bundled inventory command against the login account's `~/.codex/agent-coordinator` returned 16 managed servers. Source tracing also showed that both automatic origin discovery and default persistent source-configuration ownership derive from the remapped user-home or user-domain location, so the blast radius includes discovered sources and Manage Sources settings rather than one list view.

## Changes

Product: Automatic Board discovery, helper lookup, and default source-configuration ownership now use the effective POSIX account home. Existing remapped Foundation configuration is migrated only when the account-owned pair is absent; primary/last-known-good writes are serialized across Board processes and roll back as one pair. Physical coordinator homes are canonicalized so aliases deduplicate and a disabled alias suppresses the same automatic source. The coordinator CLI now also defaults through `getpwuid(geteuid())`, so two same-UID Codex/Parall runtimes share one home and lock despite different `HOME` values. Explicit coordinator-home overrides remain authoritative. Different effective UIDs remain isolated and reject foreign-owned/non-private homes; separate accounts require disjoint port ranges or an authorized broker for host-wide uniqueness.
Prevention: Inventory telemetry now binds each loaded source fingerprint to its decoded server count. The signed launch gate discovers every automatic Codex, Claude, and Parall source, preflights each through the packaged helper without Docker sampling, and rejects a missing source, a source-count mismatch, filtered-away managed servers, stale process identity, or a failed refresh. Detector coverage includes the real failure shape (correct populated source decoded as zero), a second Codex-instance source being dropped, an intentionally disabled source, and a genuinely empty source. Center-pane sizing now has a realistic 1180-point must-catch fixture and wider-window false-positive controls.

## Verification

Original path: Before the repair, the signed Board reproduced the user's zero-server screen while reporting one loaded source. With the repaired CLI and the same remapped shell environment, direct inventory now resolves `/Users/holyglory/.codex/agent-coordinator` and returns 16 managed servers instead of resolving the empty Parall runtime home.
Checks: Repository freshness is `current` against `origin/main`. All 88 DevOps Board Swift tests pass, including the real two-process configuration transaction, two-Parall discovery, source-alias controls, polling cadence, and retained-content refresh behavior. The launch-readiness detector passes realistic recall/control fixtures for all-source identity and per-source counts. The coordinator multi-runtime suite passes normal and optimized modes, and the full coordinator self-test passes normal, optimized, and standalone-copy execution. Repository validation, boundary/history guards, skill-link verification, and 120 Console tests pass. The native 1180-point center-pane render was inspected and keeps the toolbar, filters, tabs, empty state, and footer inside the pane.
Residual risk: Closure still requires a clean committed package, a fresh signed launch whose telemetry matches every packaged-helper source count, and sustained health across subsequent automatic refresh intervals. Cross-user host-wide port uniqueness is intentionally not promised without an authorized broker or disjoint ranges.
