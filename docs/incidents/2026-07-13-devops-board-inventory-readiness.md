# DevOps Board inventory readiness incident

## Outcome

Status: fixed

The signed Board now loads every discovered coordinator source and reports all 19 managed servers as visible to the Dev Servers presentation. Automatic refreshes complete as short bounded bursts, retain the loaded presentation, and return to idle instead of leaving the Board continuously “updating.”

## Cause

Class: implementation
Confidence: confirmed
Request: DevOps Board must run against the user's real coordinator source, keep refreshes efficient, expose working source details, and truthfully show the managed servers owned by that source.
Immediate cause: Two defects occurred in sequence. First, the Board's coordinator wrapper applied a 1,048,576-byte command-output ceiling to a valid 3,271,993-byte inventory and discarded the truncated snapshot. After that limit was fixed, automatic source discovery still derived the user home from `FileManager.default.homeDirectoryForCurrentUser`; under the Parall launch environment, `HOME` and `CFFIXED_USER_HOME` remapped that value to a Parall-managed runtime home. That runtime home contained a valid coordinator source with zero managed servers, while the login account's `~/.codex/agent-coordinator` contained the expected managed-server records.
Why missed: Native tests injected explicit homes, and the launch detector accepted any stable app process that logged at least one syntactically loaded source. Its `loaded=1 total=1` marker carried neither the resolved source-home identity nor resource counts, so a valid but empty source under the wrong owner passed. The prior delivery therefore incorrectly described that marker as proof that the real coordinator source was loaded; the user's follow-up screen contradicted that claim.
Evidence: The original unavailable state reproduced the 1 MiB truncation against a 3,271,993-byte valid account inventory. For the remaining no-server state, the signed app process inherited Parall-remapped `HOME` and `CFFIXED_USER_HOME`; automatic discovery resolved the corresponding runtime coordinator source, whose inventory command succeeded with zero managed servers. Running the same bundled inventory command against the login account's `~/.codex/agent-coordinator` returned 16 managed servers. Source tracing also showed that both automatic origin discovery and default persistent source-configuration ownership derive from the remapped user-home or user-domain location, so the blast radius includes discovered sources and Manage Sources settings rather than one list view.

## Changes

Product: The completed product change implemented effective-POSIX-account ownership for automatic Board discovery, helper lookup, and default source configuration. Existing remapped Foundation configuration is migrated only when the account-owned pair is absent; primary/last-known-good writes are serialized across Board processes and roll back as one pair. Physical coordinator homes are canonicalized so aliases deduplicate and a disabled alias suppresses the same automatic source. The coordinator CLI now also defaults through `getpwuid(geteuid())`, so two same-UID Codex/Parall runtimes share one home and lock despite different `HOME` values. Explicit coordinator-home overrides remain authoritative. Different effective UIDs remain isolated and reject foreign-owned/non-private homes; separate accounts require disjoint port ranges or an authorized broker for host-wide uniqueness.
Prevention: Completed prevention work added telemetry that binds each loaded source fingerprint to its decoded server count. The signed launch gate discovers every automatic Codex, Claude, and Parall source, preflights each through the packaged helper without Docker sampling, and rejects a missing source, a source-count mismatch, filtered-away managed servers, stale process identity, or a failed refresh. Detector coverage includes the real failure shape (correct populated source decoded as zero), a second Codex-instance source being dropped, an intentionally disabled source, and a genuinely empty source. Center-pane sizing now has a realistic 1180-point must-catch fixture and wider-window false-positive controls.
