# DevOps Board repository identity and layout incident

## Outcome

Status: fixed

The signed Board now renders one project per validated local Git worktree, renders Nevod once, keeps unattributed evidence in one non-actionable Unassigned Resources group, and keeps the center toolbar, scrollable body, Activity surface, and status footer inside the middle pane.

## Cause

Class: implementation
Confidence: confirmed
Request: DevOps Board must show one project for one local repository, show Nevod once, keep non-repository resources out of the project tree, and keep the center panel fully visible below the window toolbar. The user also requested a fundamental account of how repositories, servers, Docker resources, and related state are stored and how that design should be corrected.
Immediate cause: The Board discovers the account coordinator plus two legacy Parall coordinator homes. Each source independently returns the same host-global Docker inventory. The Board deduplicated physical containers, but keyed `project_usage` and `ProjectGroup` by `(source origin, usage_key)`, so one canonical Git root became one project per source and the non-selected copies could become empty shells. The Project Load strip consumed those source-scoped rows directly, producing three Nevod entries. The coordinator also emits `name:*` rows with no project path for unattributed containers, and the Board presented each row as a project. Separately, an intrinsically taller center view was centered inside an exact-height frame and clipped, removing its top and bottom edges.
Why missed: The multi-source regression fixture asserts source separation at the project level to avoid native server-ID collisions instead of modeling one repository with source-scoped resource references. This contradicts the recorded 2026-07-07 repository-grouping decision. The previous narrow-pane check covered horizontal width but did not exercise the real restored-window top-boundary failure shown by the user. Launch verification proved source and server counts, not canonical repository cardinality, unassigned-resource behavior, or top-edge visibility.
Evidence: Direct real inventory on 2026-07-13 returned source contributions of 16/3/0 presented servers, while every source returned the same 15 Docker containers and seven PostgreSQL projections. Their 11/9/9 project-usage rows total 29 rows for 13 distinct usage keys; only ten are path-backed, nine are currently valid Git roots, and one is a missing historical path. All three sources returned the same redacted Nevod path identity with two containers and approximately 3.73 GB. All three returned `name:aicursegmailcheck` with no project path. Persisted stores were 8,211,325, 1,550,427, and 496,675 bytes; the current two-pass inventory path rewrites approximately 39.13 MiB of JSON payload per Board refresh and observes the same Docker engine once per source. The 2346x1698 user screenshot has SHA-256 `ab7bf1bbb8ba49c53beecd61ecc043a42339ed355400483262f05f06794b9e8e` and visibly shows the toolbar/top of the center panel outside the content boundary, repeated left-tree projects, and three Nevod load rows. The detailed source trace and proposed target are recorded in [`repository-catalog-and-state-store.md`](../architecture/repository-catalog-and-state-store.md).

## Changes

Product: `RepositoryCatalog` now resolves only existing canonical Git worktree roots as projects, builds one aggregate across every source observation, reconciles Docker by immutable container ID and processes by PID, and retains source-qualified resource identities for safe control routing. Missing, non-Git, name-only, and conflicting evidence appears once under Unassigned Resources instead of manufacturing projects. Whole-repository mutations require one proven controller covering the complete runtime; ambiguous membership or active endpoints remain visible, mark health non-nominal, and block before external action. Logical server/container/database selection survives a change in which source supplies the actionable representative. The center pane is top-aligned with fixed toolbar/Activity/status chrome and a vertically scrollable variable body.
Prevention: Native production-view regressions now exercise dense 524x760 and 784x900 layouts, the real center-only upward crop, and intentional inner-scroll/empty-body controls. Repository tests cover three-source Nevod, aliases/nested roots, same-name distinct roots, name-only evidence, duplicate state rows, conflicting cross-repository claims, physical metric deduplication, source-switch selection, and fail-closed action routing. Launch telemetry and readiness reject repository/group cardinality divergence or more than one Unassigned group. Canonical snapshots are generated by an explicit SwiftPM test target, bound to all renderer and repository-catalog sources, and checked for semantic chrome with realistic recall and dark-UI false-positive controls. The normalized SQLite store, single host observer, same-UID legacy importer, and optional cross-user broker remain an explicit proposal rather than implemented behavior.

## Verification

Original path: Post-fix, I re-ran and verified the original signed-Board path from commit `5f208ae` through the packaged-helper readiness gate with all three real sources. Four consecutive refresh markers retained raw per-source server counts `0`, `16`, and `3`, while the logical Board presentation retained `managed=19 visible=19 repositories=9 repository_groups=9 unassigned_groups=1`. A 75-second process-tree sample covered two automatic refresh bursts: helpers rose briefly to six children, then returned to zero; the app spent the intervening and final samples at `0.0%` CPU with no children. The native production hierarchy at 524x760 and 784x900 keeps both fixed edges and usable body content visible; all four current-source canonical Board/menu artifacts were independently inspected without top cropping or overlap.
Checks: All 124 native tests pass with one intentional snapshot-regeneration skip. The selected canonical renderer test passes 1/1, and the unflagged verifier accepts all four PNG/provenance pairs. The full non-native repository gate passes the 120-test Console suite, coordinator and backup self-tests in normal/optimized/standalone modes, launch-readiness recall/control tests, snapshot recall/control tests, public/reachable-history boundaries, package self-test, and static compilation. Same-UID multi-runtime tests pass in normal and optimized modes, proving two remapped runtimes share one private home and lock; the different-UID fixture proves distinct account homes and rejects foreign state access before mutation. Packaging verified the app signature and exact build-input/runtime-helper hashes.
Residual risk: The compatibility catalog corrects identity, presentation, aggregation, and action safety, but each legacy source still performs its own JSON inventory writes and observes the same Docker engine. The idle-after-completion scheduler prevents continuous or overlapping refresh work, yet short refresh CPU/I/O bursts remain. The proposed normalized SQLite/WAL store, pure inventory read, same-UID legacy import, one observer per Docker engine, and authorized cross-user broker are not implemented. Different Linux users therefore still require disjoint port ranges unless an authorized broker is deployed.
